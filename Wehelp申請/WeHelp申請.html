個人簡介和申請動機

我是沐勁宇，今年 25 歲，畢業於國立體育大學體育推廣學系。畢業後曾從事百貨營業管理，目前於蝦皮智取店服務專線擔任客服。在工作中，需要釐清使用者問題、判斷問題來源（系統 / 機台 / 訂單），並提供解決方案，這樣的經驗讓我逐漸習慣用邏輯方式思考與解決問題。

從小我就對電腦相關事物有興趣，國小時曾接觸過 Dreamweaver 課程，對網頁的運作方式留下初步印象。雖然當時沒有持續深入，但在去年重新思考職涯方向時，我回想起這段經驗，並開始實際投入程式學習， 也逐漸發現自己對於透過邏輯解決問題的過程感到有成就感，也開始確認這是自己適合長期發展的方向，因此決定投入更多時間學習相關技術。目前已經開始學習 HTML、CSS 與 JavaScript、Python，並嘗試實作基礎的網頁與互動功能。在自學的過程中，我發現自己在過程中容易缺乏方向與系統性的規劃，因此希望透過有任務導向的學習環境來強化自己的能力。以實作為導向的方式也符合我希望培養的能力，我期望在這個高強度的學習中成長，因此我希望能加入Wehelp，朝軟體工程師的目標邁進。

曾經做過哪些軟體工程技術相關的學習？

在接觸前端開發之前，我先從 Python 開始入門，透過 YouTube 教學影片自學基礎語法，包含條件判斷、迴圈與函式等概念。在學習過程中，我曾實作簡單的「提款機系統」與「遊戲對戰系統」，透過程式模擬邏輯流程，讓我對程式如何解決問題有初步的理解，也從中獲得成就感。 在確認轉職方向後，我開始專注於前端技術，自學 HTML、CSS 與 JavaScript。從基礎的 HTML 標籤與 CSS 區塊模型、選擇器與版面配置開始，並實作如「NBA 球員卡」與「品牌網站」等切版練習，熟悉網頁結構與版面設計。

在學習 JavaScript 後，我進一步嘗試加入互動功能，練習 DOM 操作與事件處理，例如建立資料新增與顯示功能，透過 addEventListener 綁定事件，使網頁具備基本互動性。同時也練習使用陣列方法（如 filter、includes）來實作關鍵字搜尋功能，後續也寫了一個Todo list 的專案來加強自己的實作能力。 在實作過程中，我曾對事件處理的寫法感到困惑，例如一開始不理解為什麼 addEventListener 傳入函式時不需要加括號，導致函式在頁面載入時就被執行，而非在事件觸發時執行。此外，在使用 filter 進行資料篩選時，也曾因為不熟悉其回傳新陣列的特性，以及對箭頭函式語法不熟，導致搜尋功能無法正確實現，透過查詢資料與反覆測試，我逐步理解其運作方式並成功解決問題。這樣的經驗讓我逐漸建立起以邏輯拆解問題並自行尋找解法的能力。

如果參與這個訓練，會怎麼安排學習時間？

若有機會參與訓練，我預計每天投入至少 6小時以上的時間在學習與實作上，約2小時用來學習當週需要的技術與觀念，並開始動手實作，其餘時間則會持續撰寫程式、完成每週任務，以及處理在實作過程中遇到的問題。

若在學習過程中遇到困難，會優先透過查詢資料與自行嘗試解決，並在需要時於社群中提問，確保問題能被釐清。在週末時間，除了持續完成進度，也會用來回顧整週學習內容，檢視自己的理解程度與不足之處，並調整整接下來的學習方式與節奏。我也會以穩定且持續的方式投入訓練，確保能跟上每週進度，並逐步累積實作能力。

請簡述對於軟體工程的領域偏好和職涯想像

在目前的學習過程中，我對前端開發較有興趣。透過 HTML、CSS 與 JavaScript 的學習與實作，我嘗試完成網頁版面與基本互動功能，當能夠透過程式讓畫面產生變化，並正確回應操作時，會讓我有成就感，也讓我對前端開發產生興趣。

未來我希望能先以前端工程師為目標，持續加強 JavaScript 的基礎能力，並進一步學習更完整的前端開發工具與框架（如 React），提升開發效率與程式架構能力。同時，在具備一定基礎後，也希望接觸後端技術，例如 API 設計與資料庫操作，讓自己能夠理解整體系統的運作方式，逐步朝全端工程師的方向發展。在職涯規劃上，我希望能進入實際的開發團隊，透過實務經驗持續累積能力，從基礎功能開發做起，逐步成為能獨立負責功能並解決問題的工程師。

請描述一件產生明顯負面情緒的經歷，如何處理該情緒？

在蝦皮服務初期，我曾處理過一起極為激動的智取店包裹遺失客訴，用戶因已付款卻面臨開櫃空櫃，用戶很激動的表示他已經付款為什麼包裹沒拿到，我先是安撫用戶情緒，說明後續請門市人員協尋到他的包裹會協助重新上架並通知用戶來領取，後續沒收到包裹也可以在APP申請退貨退款，用戶表示不接受這個處理方式，要求當下賠償以及主管出面處理，因公司有SOP所以無法協助用戶進行轉接指定人員，後續由於權限的關係，無法進一步協助用戶，剛剛的訴求會登記下來通報給相關單位處理。

因為這是第一次遇到這樣子的經驗，所以在通話結束後我的情緒明顯受到影響，結束後我給自己10分鐘的時間到休息區調整情緒，確保自己不會因為這件事情影響到後續的接聽電話，並且也同步和主管與同事確認流程的正確性，以及話術上的一些技巧，這讓我了解到必須更堅定自己權限內能處理的事情，不被用戶影響而自己心虛，這讓我在後續接聽到類似客訴案件時，因清楚自己該做什麼事而產生心理上的穩定感，不再輕易被用戶的情緒干擾，進而能更專注於解決問題本身。

關於這份申請網頁，分享一個開發時的技術心得

在開發這個網頁時，HTML 的部分我主要想要有「導覽列」、「內容區塊」、「頁尾」，導覽列讓我可以點擊時對應到相對的區塊閱讀，所以 nav 建立時我選擇使用 Flexbox 進行排版，這讓我能使用 justify-content 將選單置中，並透過 gap 屬性精確控制項目間距，導覽列的項目透過 padding 控制內距、border-bottom 先新增透明的底線才不會因為後續 hover 加入底線導致跳動，後面我在導覽列透過 CSS 的 hover 提升了互動性。最後是我想在瀏覽時固定 header 區塊，所以我使用了 position: sticky 與 top: 0，確保在頁面捲動時能始終維持在視窗頂部。這讓使用者能隨時切換章節，同時 z-index: 1000 能確保導覽列具備最高層級，不被下方內容遮蓋。最後加入 box-shadow 讓 header 下方與內容產生不同層的邊界。

在處理網頁主體內容「main」時，我主要關注的是視覺聚焦與閱讀舒適度。在 main 容器上設定寬度為 90%，也用 max-width 限制容器最大的寬度，並利用 margin: auto 達成水平置中，規範了資訊呈現的邊界。透過 padding 避免文字緊貼邊界，同時利用背景色定義出內容區塊感。 為了讓資訊呈現條理分明，接下來我也利用「section」來切分每一個題目。因標題的 div 預設會佔滿一整行，所以設定 display: inline-block 讓背景顏色能根據文字長度自動縮放區塊，並透過 text-underline-offset 讓底線不會緊貼文字；內文的部分我在容器上使用 gap 來控制不同段落的間距，內容的部分則使用 line-height 控制行高。 「Footer」的部分我主要分成「聯絡資訊」與「相關連結」，在最外層容器 footer 上使用 display: flex 讓裡面兩個部分水平排列，裡面標題與內容的部分則利用 flex-direction 控制垂直排列，讓資訊條列在版面上。為了一眼就能區分標題與內容，內容部分利用 margin-left 做縮排設計。 版面主要調整完之後，我決定開始寫錨點連結的部分，利用 scroll-behavior: smooth 優化成動態的平滑捲動。在往下滑動的時候，我發現導覽列會擋住該段落的標題，於是我利用 scroll-margin-top 來建立預留的緩衝區，解決了這個問題。接下來我想讓滑動到該區域時能做高亮顯示，於是我運用了 :target 結合自定義的 @keyframes highlight 動畫。當透過導覽列點擊特定章節時，目標區塊會產生短暫的背景高亮閃爍。

接下來就是寫「RWD 響應式網頁設計」，首先在「max-width: 1024px」，我將 main 容器的寬度與 padding 做調整，確保內容仍能保有足夠的橫向空間，也適度地縮小標題與內容區塊元素的體積，並增加內文的行距，在較小的螢幕上能有效避免段落擠壓。 在「max-width: 850px」時，導覽列面臨擠壓的問題。我選擇了「橫向滑動」處理，透過 justify-content: flex-start 與 white-space: nowrap 讓導覽連結維持在同一行，避免因空間不足而產生的雜亂換行，並設定 overflow-x: auto 讓選單在超出螢幕寬度時，能自動產生滑動功能而不破壞版面。 最後處理的是「max-width: 600px」，在行動裝置上我希望將導覽列做垂直排列，於是我新增了一個選單的 div 讓我可以實作點擊展開的機制。透過在 HTML 中加入 onclick="toggleMenu()"，讓我在點擊選單按鈕時，便會觸發定義的函式。在 toggleMenu() 函式中，我選用 classList.toggle('active') 進行類別切換，透過增刪 Class 來切換 CSS 的顯示狀態：平時將 nav 設為 display: none 隱藏，當帶有 .active 類別時切換為 display: flex 展開，從而達成選單收摺的效果。內容 main 的區塊我也適當地調整 padding 與 width 使內容能撐滿畫面，footer 的部分我則改將相關連結做垂直排列。

透過此專案，我掌握了 HTML 語意化結構與 CSS 響應式佈局的實務應用。我利用 CSS 解決不同裝置間的適配問題，從斷點的精確微調到 Flexbox 的靈活運用，確保了網頁在電腦與行動端皆能有良好的排版。此外，我也透過微量的 JavaScript 實作狀態切換。這次開發不僅讓我學會如何撰寫代碼，更讓我掌握了如何解決問題，這次遇到了導覽遮擋、區塊溢出處理等實際的介面挑戰，透過這個網頁，我已將 HTML 與 CSS 的理論知識轉化成實作經驗，為後續進階開發奠定了穩固基礎。

如何看待人工智慧的發展對軟體工程師的影響？

我認為AI工具極大降低了語法實作的門檻，現在能透過 AI 快速生成基礎框架，我們必須具備更深厚的底層邏輯才能判斷 AI 生成內容的安全性與效能。對我來說AI可以加速我的學習，能即時解決我遇到的各種問題，讓我能更專注於思考如何透過程式解決真正的問題。在未來，能理解商業需求、並將複雜問題拆解的工程師，將會有很大的價值，總結來說我覺得AI是一個很好的工具，但是熟練的基本功才是穩定工具的根本，我希望透過訓練能建立扎實的工程師思維，運用AI創造出有價值的成果、產品。

其他想要對我們說的事情？

在評估眾多轉職管道後，我覺得 WeHelp 是最符合我的環境。我瞭解軟體工程師需要不斷的「解決問題」。我認同 WeHelp 的「不給標準答案，而是引導學員自主找尋解法」的教學核心，在過程中實作的比例應該佔大多數，在準備這份申請書的期間，我也透過自學完成了申請網頁，並在過程中享受那種為了釐清 CSS 佈局細節而反覆實驗的過程。 此外，我明白工程師在開發與 Debug 時常會面臨巨大的壓力，但我具備了穩定的情緒並有尋求解決方法的韌性。也準備好投大量的時間，對我而言，這不僅是一次轉職，更是一次自我的挑戰。我希望在 WeHelp 的高強度磨練下，轉化為一名真正具備實戰開發能力、能產出實質價值的軟體工程師。
